## Task Breakdown

### Task 1: Define FailureCode Taxonomy

**What:** Create `src/dm_bot/testing/failure_taxonomy.py` — standardized failure classification enum.

**Why:** Every scenario failure must be classified with a standard code so patterns can be tracked across runs.

**Enum:**
```python
class FailureCode(str, Enum):
    PHASE_TRANSITION_MISMATCH = "PHASE_TRANSITION_MISMATCH"
    COMMAND_GATE_FAILURE = "COMMAND_GATE_FAILURE"
    REVEAL_POLICY_VIOLATION = "REVEAL_POLICY_VIOLATION"
    VISIBILITY_LEAK = "VISIBILITY_LEAK"
    RULE_RESOLUTION_MISMATCH = "RULE_RESOLUTION_MISMATCH"
    SESSION_STATE_MISMATCH = "SESSION_STATE_MISMATCH"
    PERSISTENCE_RECOVERY_FAILURE = "PERSISTENCE_RECOVERY_FAILURE"
    STREAM_RECOVERY_FAILURE = "STREAM_RECOVERY_FAILURE"
    INTENT_ROUTING_MISMATCH = "INTENT_ROUTING_MISMATCH"
    NARRATION_CONTRACT_VIOLATION = "NARRATION_CONTRACT_VIOLATION"
    CONCURRENCY_INVARIANT_FAILURE = "CONCURRENCY_INVARIANT_FAILURE"
    SCENARIO_TIMEOUT = "SCENARIO_TIMEOUT"
    UNSUPPORTED_TEST_CONTRACT = "UNSUPPORTED_TEST_CONTRACT"
```

**Failure record shape:**
```python
@dataclass
class Failure:
    code: FailureCode
    message: str
    step_index: int
    expected: Any
    actual: Any
    artifact_pointer: str | None = None
```

**How:**
1. Create `src/dm_bot/testing/failure_taxonomy.py`
2. Define `FailureCode` enum with all 13 types
3. Define `Failure` dataclass
4. `ScenarioResult.failure: Failure | None` — replace any existing failure field

**Acceptance:**
- All 13 FailureCode values exist as string enums
- `Failure` dataclass has all required fields
- `ScenarioResult` uses `Failure` for failure field

### Task 3: Write Visibility Leak Scenarios (contract/visibility/)

**What:** Create YAML scenarios in `tests/scenarios/contract/visibility/` that verify gm_only state never reaches player output.

**Why:** This is the core "visibility leak" contract test.

**Scenarios to write:**

**`scen_no_gmonly_to_player.yaml`**
- Setup: session with 1 KP + 1 player
- KP sets a gm_only field on an entity
- Player takes a turn
- Assertion: `player_forbidden` — player output must NOT contain gm_only field identifiers (e.g., internal clue IDs, internal notes)
- Expected: PASS (no leak)

**`scen_gmonly_reaches_kp.yaml`**
- Same setup
- KP takes turn
- Assertion: `kp_must_include` — KP output MUST include gm_only field (KP has full visibility)
- Expected: PASS

**How:**
1. Create `tests/scenarios/contract/visibility/` directory
2. Write YAML scenarios using DSL from E70
3. Each scenario: 5-10 steps, minimal viable setup
4. Assertions use `visible.player_forbidden` and `visible.kp_must_include`

**Acceptance:**
- `uv run python -m dm_bot.main run-scenario --suite contract --tags visibility` runs all
- All visibility scenarios pass with `VISIBILITY_LEAK` = none found

### Task 4: Write Reveal Policy Scenarios (contract/reveal/)

**What:** Create YAML scenarios verifying clue reveal gates are enforced.

**Why:** Reveal gates should only open when conditions are met. Premature reveal = bug.

**Scenarios to write:**

**`scen_clue_requires_investigation.yaml`**
- Setup: session with clue that requires "examining door" action to reveal
- Player messages WITHOUT the prerequisite action
- Assertion: clue NOT revealed to player
- Player messages WITH prerequisite action
- Assertion: clue revealed to player

**`scen_wrong_path_no_premature_reveal.yaml`**
- Setup: fuzhe_mini with reveal gate
- Player takes wrong path (doesn't trigger condition)
- Assertion: reveal gate NOT opened

**How:**
1. Create `tests/scenarios/contract/reveal/` directory
2. Write scenarios using `fuzhe_mini` adventure
3. Use `narration.must_not_reveal_hidden_truth: true` where applicable

**Acceptance:**
- Wrong-path scenario: reveal gate stays closed (PASS)
- Investigation scenario: gate opens after prerequisite (PASS)

---

### Task 5: Write AI Contract Scenarios (contract/ai_contract/)

**What:** Create YAML scenarios verifying router/narrator packet structure and prompt isolation.

**Why:** AI contract = router input packet correct, narrator prompt has no forbidden state, audience split works.

**Scenarios:**

**`scen_router_packet_structure.yaml`**
- Player sends a message
- Assertion: `router_requests` captured by FastMock contains expected `mode`, `tool_calls`, `state_intents`
- mode must be "dm" or "narrate" or "combat" — not random string
- tool_calls must be list (even if empty)

**`scen_narrator_no_hidden_in_prompt.yaml`**
- KP sets gm_only state on a clue
- Narrator is called
- Assertion: `narrator_requests` captured by FastMock — the prompt/packet does NOT contain gm_only identifiers
- Check: internal clue IDs, gm_only flags, hidden story node references

**`scen_audience_split_correct.yaml`**
- KP action generates both public and KP-private narration
- Assertion: FastMock captured outputs have correct `audience` tags
- `audience: "kp"` for KP-only content
- `audience: "public"` for shared content

**How:**
1. Create `tests/scenarios/contract/ai_contract/` directory
2. Use `model_mode: fake_contract`
3. FastMock already captures `router_requests` / `narrator_requests` — inspect after run
4. Assertion: inspect captured request payloads from `driver.snapshot_state()`

**Acceptance:**
- Router packet has valid `mode` field
- Narrator prompt has no `gm_only` strings
- Audience split is correct in captured outputs

### Task 6: Record VCR Cassettes for `recorded` mode

**What:** Record at least one real router + narrator response as VCR cassette.

**Why:** `model_mode: recorded` needs cassettes to replay. None exist yet.

**How:**
1. Set up real Ollama connection (check `tests/conftest.py` for existing VCR config)
2. Run one scenario in `live` mode with recording enabled
3. Cassette saved to `tests/cassettes/<scenario_id>/<step_name>.yaml`
4. Add at least:
   - One router call cassette (structured response)
   - One narrator call cassette (text response)
5. Configure VCR filters to redact API keys / sensitive data

**Note:** This requires Ollama running. If Ollama not available, document as manual step and skip with comment in code.

**Acceptance:**
- `tests/cassettes/` directory created with at least 2 cassette files
- Cassettes are valid YAML/JSON replayable by VCR
- `model_mode: recorded` scenario uses cassette if exists, errors helpfully if not
