## Task Breakdown

### Task 1: Write `acceptance/scen_session_happy_path.yaml`

**What:** Full session lifecycle scenario — the definitive "can we run a session?" test.

**Steps:**
1. KP `bind_campaign(camp_happy)`
2. P1 `join_campaign(camp_happy)`
3. P2 `join_campaign(camp_happy)`
4. KP `load_adventure(fuzhe_mini)`
5. P1 `select_profile(p1_default)` + `ready()`
6. P2 `select_profile(p2_default)` + `ready()`
7. KP `start_session()` → verify: phase → `onboarding`
8. KP `complete_onboarding()` → verify: phase → `scene_round_open`
9. P1 `message("我检查门把手")` → round open
10. P2 `message("我查看窗户")` → round open, pending
11. KP `resolve_round()` → verify: phase → `scene_round_resolving`
12. KP `next_round()` → verify: phase → `scene_round_open`

**Assertions:**
```yaml
phase_timeline:
  - lobby
  - awaiting_ready
  - awaiting_admin_start  (KP start_session)
  - onboarding
  - scene_round_open
  - scene_round_resolving
  - scene_round_open  (next_round)
final_phase: scene_round_open
state:
  campaign_members: 2
  no_duplicate_members: true
visible:
  public_must_include:
    - "已就位"
```

**Acceptance:** Passes with full phase timeline correct.

### Task 2: Write `acceptance/scen_fuzhe_15turn.yaml`

**What:** 15-turn run through `fuzhe_mini` with seeded dice (deterministic).

**Why:** This is the "real" acceptance test — long enough to exercise triggers, reveals, consequences.

**Approach:**
- Use `fuzhe_mini` (4-node adventure from E69)
- `dice_seed: 42` — deterministic
- 15 turns: player actions exploring all 4 nodes
- Key assertions at each phase: trigger fires, clue revealed at correct time, consequence state updated

**Assertions:**
- Phase timeline covers all nodes
- At least 3 triggers fire in expected order
- At least 1 reveal gate opens at correct time (not premature)
- No visibility leaks
- End state reflects all consequences

**Note:** If `fuzhe_mini` (E69) is not ready, use `fuzhe.json` with explicit 4-node scope.

**Acceptance:** 15 turns complete, all assertions pass, deterministic (same seed = same result).

### Task 3: Write `recovery/scen_crash_recovery.yaml`

**What:** Mid-scenario crash + restart, verifying state is recoverable.

**Steps:**
1. Run happy path up to `scene_round_open` with 2 members
2. `simulate_crash()` — abruptly stop runtime (no graceful shutdown)
3. `restart_runtime()` — reload from `PersistenceStore`
4. Verify:
   - Campaign bindings intact
   - Member list correct
   - Phase restored to `scene_round_open` (or nearest safe phase)
   - No duplicate members
5. Continue session after recovery
6. Complete successfully

**Assertions:**
```yaml
persistence:
  restart_recoverable: true
state:
  no_duplicate_members: true
  campaign_members: 2
```

**Key test:** `driver.simulate_crash()` + `restart_runtime()` round-trips correctly.

**Acceptance:** Session resumes correctly after simulated crash, no data loss.

### Task 4: Write `recovery/scen_stream_interrupt.yaml`

**What:** Streaming narration interrupted mid-stream, resume contract verified.

**Steps:**
1. Start a turn that produces streaming narration
2. `simulate_stream_interrupt()` — interrupt mid-stream

4. Verify: final narration is complete (not truncated at interrupt point)
5. Verify: state is consistent with completed narration

**Assertions:**


**Note:** Requires `FakeStreamingTransport` from E69 (FakeMessage.edit() support).

**Acceptance:** Interrupted stream resumes correctly, no state corruption.

### Task 5: Write `chaos/scen_chaos_lobby.yaml`

**What:** 5 concurrent users all joining, readying, and acting simultaneously.

**Why:** Concurrency stress test — no duplicate members, correct phase under load.

**Steps:**
1. 5 players all `join_campaign` simultaneously
2. All 5 `select_profile` + `ready` concurrently
3. KP `start_session` with all 5 ready
4. All 5 submit actions in same round (fast sequential, same timestamp)
5. KP `resolve_round`
6. Verify: exactly 5 members, no duplicates
7. Verify: all 5 actions collected
8. Verify: `get_pending_actions` count = 5

**Assertions:**
```yaml
state:
  campaign_members: 5
  no_duplicate_members: true
visible:
  kp_must_include:  # all 5 pending
    - "待提交: 5"
```

**Acceptance:**
- No duplicate members under concurrent join
- All 5 actions collected
- Correct phase transitions throughout

### Task 6: Configure CI Execution

**What:** Ensure all scenario suites run in CI without live Discord, Ollama, or GUI.

**Why:** Acceptance gate must be CI-automatable.

**Requirements for CI:**
1. `uv run pytest tests/scenarios/ --` runs all scenarios (or `uv run python -m dm_bot.main run-scenario --all`)
2. Headless: no Discord connection, no browser, no GUI
3. `model_mode: fake_contract` by default — no live AI
4. Each scenario: timeout 60s max
5. Artifacts written to `artifacts/scenarios/` (gitignored)

**How:**
1. Add `pytest.ini` or `pyproject.toml` config for `tests/scenarios/`:
   ```toml
   [tool.pytest.ini_options]
   testpaths = ["tests/scenarios"]
   asyncio_mode = "auto"
   timeout = 60
   ```
2. Or use `conftest.py` in `tests/scenarios/` that imports from root `conftest.py`
3. Ensure `tests/scenarios/` has `conftest.py` that imports fixtures from `tests/conftest.py`
4. Add `artifacts/` to `.gitignore`
5. Verify: `uv run pytest tests/scenarios/ -v` runs all 6 scenarios (smoke + happy + fuzhe + crash + stream + chaos)

**CI command:**
```bash
uv run pytest tests/scenarios/ -v --tb=short
```

**Acceptance:**
- CI runs all scenario suites headlessly
- Exit code 0 (all pass) or 1 (any fail)
- Artifacts generated for failures
- No live network calls

### Task 7: Migrate Existing E2E Tests (Optional/Candidate)

**What:** Evaluate migrating `test_e2e_15turn_scenario.py` and `test_chaos_lobby_stress.py` to YAML scenarios.

**Why:** Reduce duplication — these tests follow a scenario-like pattern already.

**Decision criteria:**
- If YAML version covers same ground as existing pytest version → migrate
- If YAML version is harder to maintain than pytest version → keep pytest
- Don't migrate just for the sake of migration

**How:**
1. Run existing tests alongside new YAML scenarios
2. If coverage overlaps ≥80% → migrate
3. If coverage gaps exist → keep pytest version

**Acceptance:** At minimum, no coverage regression. If migrated, fewer total test files.
