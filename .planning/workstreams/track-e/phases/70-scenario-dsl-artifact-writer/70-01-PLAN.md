# Phase 70 Plan: Scenario DSL + ArtifactWriter

**Phase:** E70-01
**Milestone:** vE.2.2 — 统一 Scenario-Driven E2E 验证框架
**Goal:** Define structured YAML scenario DSL, implement ArtifactWriter, add `run-scenario` CLI command, define `model_mode` strategy and initial state setup.
**Depends on:** E69 (RuntimeTestDriver + StepResult contract)

---

## Context

E69 creates `RuntimeTestDriver` and `ScenarioRunner` with a basic YAML loader. E70 formalizes the DSL format, implements full artifact output (json + md), adds the CLI entry point, and defines how `model_mode` and initial state setup work across all scenario types.

**E70 does NOT implement scenarios** — those are E71 (contract) and E72 (acceptance).

## Task Breakdown

### Task 1: Define and Implement Scenario DSL (YAML Format)

**What:** Create `src/dm_bot/testing/scenario_dsl.py` — the canonical YAML schema for scenarios, with parser and validation.

**Why:** All scenarios (acceptance, contract, chaos, recovery) must use the same structured format.

**YAML Schema:**
```yaml
id: scen_<name>              # unique, kebab-case
title: "<Human readable>"    # display name
tags: [acceptance, recovery]  # suite membership
mode: acceptance             # acceptance | contract | chaos | recovery
# mode affects: assertion strictness, artifact naming

actors:
  kp:
    kind: admin
    user_id: u_kp
  p1:
    kind: player
    user_id: u_p1
  p2:
    kind: player
    user_id: u_p2

fixtures:
  adventure: fuzhe_mini        # module name (optional)
  dice_seed: 42               # int, for deterministic runs (optional)
  model_mode: fake_contract    # fake_contract | recorded | live
  db_mode: temp_sqlite        # temp_sqlite | persistent (optional)

steps:
  - actor: kp
    action: command
    name: bind_campaign
    args:
      campaign_id: camp_001

  - actor: p1
    action: command
    name: join_campaign
    args:
      campaign_id: camp_001

  - actor: kp
    action: command
    name: start_session
    args: {}

  - actor: p1
    action: message
    text: "我检查门把手"

assertions:
  phase_timeline:             # optional — if present, runner verifies
    - lobby
    - awaiting_ready
    - scene_round_open

  final_phase: scene_round_open

  visible:
    public_must_include:       # output must contain these strings
      - "已加入"
    kp_must_include:
      - "待提交"
    player_forbidden:         # output must NOT contain these strings
      - "_hidden_clue_"

  state:
    campaign_members: 2
    no_duplicate_members: true

  persistence:
    restart_recoverable: true   # optional

  narration:
    must_not_reveal_hidden_truth: true  # optional
```

**How:**
1. Define `Scenario` Pydantic model (or dataclass) matching the YAML schema
2. Add `ScenarioValidator` — validates scenario structure (required fields, actor refs, action types)
3. Add `ScenarioParser` — loads and parses YAML into `Scenario` object
4. Support JSON format as alternative to YAML (same schema)
5. Add `scenario registry` — scans `tests/scenarios/` recursively, registers all scenarios

**Acceptance:**
- `ScenarioParser.parse("tests/scenarios/acceptance/scen_happy_path.yaml")` returns valid `Scenario`
- Invalid YAML (missing required fields) raises `ScenarioValidationError` with field name
- JSON variant also works
- Registry finds all scenarios in subdirectories

### Task 2: Implement ArtifactWriter

**What:** Create `src/dm_bot/testing/artifact_writer.py` — writes run records to disk.

**Why:** Each scenario run must produce machine-parseable + human-readable artifacts.

**Artifact Output Structure:**
```
artifacts/scenarios/<scenario-id>/<run-timestamp>/
  run.json              # machine-readable: metadata, steps, results, pass/fail
  summary.md           # human-readable: title, pass/fail, phase timeline, key outputs
  timeline.json         # step-by-step phase changes
  outputs_public.json   # all public-audience outputs
  outputs_kp.json       # KP-audience outputs
  outputs_player_p1.json
  outputs_player_p2.json
  state_before.json     # runtime state before first step
  state_after.json      # runtime state after last step
  failure.json          # only if failed: code, message, expected, actual, artifact_pointer
```

**`run.json` schema:**
```json
{
  "scenario_id": "scen_happy_path",
  "title": "Happy Path Session Lifecycle",
  "mode": "acceptance",
  "passed": true,
  "run_timestamp": "2026-03-30T10:00:00Z",
  "duration_ms": 1234,
  "steps": [
    {
      "index": 0,
      "actor": "kp",
      "action": "command",
      "name": "bind_campaign",
      "args": {"campaign_id": "camp_001"},
      "phase_before": "lobby",
      "phase_after": "lobby",
      "outputs": [],
      "duration_ms": 50
    }
  ],
  "phase_timeline": ["lobby", "awaiting_ready", "scene_round_open"],
  "final_state_summary": {"phase": "scene_round_open", "members": 2},
  "failure": null
}
```

**`failure.json` schema (when failed):**
```json
{
  "code": "PHASE_TRANSITION_MISMATCH",
  "message": "Expected phase 'awaiting_ready' but got 'scene_round_open'",
  "step_index": 5,
  "expected": "awaiting_ready",
  "actual": "scene_round_open",
  "artifact_pointer": "..."
}
```

**How:**
1. Create `ArtifactWriter` class
2. `write_run(scenario_id, result: ScenarioResult, artifact_dir)` — writes all artifact files
3. Use `orjson` for JSON (fast, correct)
4. Markdown summary auto-generated from `ScenarioResult` fields
5. Outputs by audience: filter `StepResult.emitted_outputs` by `audience` field
6. `state_before.json` / `state_after.json`: serialize `snapshot_state()` output
7. `failure.json`: write only when `result.failed = True`
8. Handle large streaming output: truncate `content` in artifact if > 10KB, add `"truncated": true`

**Acceptance:**
- `ArtifactWriter.write_run(...)` creates all 9+ files in artifact directory
- `run.json` is valid JSON with all required fields
- `summary.md` is readable in ~20 lines
- `failure.json` has correct `code`, `message`, `step_index`
- Large output truncated correctly

### Task 3: Add run-scenario CLI Command

**What:** Add `run-scenario` subcommand to `uv run python -m dm_bot.main`.

**Why:** Unified CLI entry for running scenarios during local dev and CI.

**CLI Usage:**
```bash
# Run single scenario
uv run python -m dm_bot.main run-scenario --scenario tests/scenarios/acceptance/scen_happy_path.yaml

# Run all in a suite
uv run python -m dm_bot.main run-scenario --suite acceptance

# Run all suites
uv run python -m dm_bot.main run-scenario --all

# Options
--write-artifacts       # Write artifacts to disk (default: True in --all, False for single if --no-artifacts)
--fail-fast            # Stop on first failure
--keep-db              # Keep temp DB files after run
--model-mode           # Override model_mode from scenario YAML
--seed                  # Override dice_seed from scenario YAML
--format               # json | md | both (default: both)
```

**How:**
1. Read `src/dm_bot/main.py` — understand CLI structure
2. Add `run_scenario` subcommand using `typer` or existing CLI framework
3. Wire up `ScenarioRunner` + `RuntimeTestDriver`
4. Output to stdout: pass/fail + phase timeline + short summary
5. Exit code: 0 if all pass, 1 if any fail

**Acceptance:**
- `uv run python -m dm_bot.main run-scenario --help` shows correct usage
- `uv run python -m dm_bot.main run-scenario --scenario tests/scenarios/acceptance/scen_happy_path.yaml` runs and prints summary
- `uv run python -m dm_bot.main run-scenario --suite acceptance` runs all acceptance scenarios
- Exit code correct (0/1)

### Task 4: Implement model_mode Strategy

**What:** Document and implement the `model_mode` parameter for scenarios.

**Why:** Three modes exist — `fake_contract` (mock), `recorded` (VCR replay), `live` (Ollama). This must be wired in RuntimeTestDriver and documented.

**Modes:**

| Mode | What it does | When to use |
|------|-------------|-------------|
| `fake_contract` | Uses `FastMock` / `StubModelClient` | Default for all E69-E72. Fast, deterministic, no network. |
| `recorded` | Replays VCR cassettes from `tests/cassettes/` | AI contract tests that need real model output shape. Cassettes don't exist yet — E71 records them. |
| `live` | Calls real Ollama API | Quality evaluation, not for CI. Requires Ollama running. |

**How:**
1. In `RuntimeTestDriver.__init__`, accept `model_mode: str` parameter
2. `fake_contract`: inject `FastMock()` into model layer — default
3. `recorded`: look for cassette in `tests/cassettes/<scenario_id>/<step_name>.yaml` — error if not found (cassettes recorded in E71)
4. `live`: use real `OllamaModelClient` — error with helpful message if Ollama not reachable
5. Document the three modes in `scenario_dsl.py` docstring

**Acceptance:**
- `RuntimeTestDriver(model_mode="fake_contract")` works without network
- `RuntimeTestDriver(model_mode="live")` errors helpfully if Ollama not running
- `model_mode` from YAML parsed correctly into driver init

### Task 5: Implement Initial State Setup Mechanism

**What:** Define and implement how each scenario gets to a known initial state.

**Why:** Without this, scenarios could have state pollution from previous runs.

**How:**
1. Each scenario run gets fresh `RuntimeTestDriver(db_path=":memory:")` — in-memory SQLite
2. Optional `fixtures.db_mode: persistent` — keeps DB at `artifacts/scenarios/<id>/db/` for debugging
3. `RuntimeTestDriver.start()` resets: fresh `SessionStore`, fresh `PersistenceStore`, no campaign state
4. `fixtures.dice_seed` passed to `SeededDiceRoller(42)` — deterministic from known seed
5. Document in `scenario_dsl.py` that `db_mode: temp_sqlite` is the default

**Acceptance:**
- Two consecutive `ScenarioRunner.run()` calls produce identical results (no state leakage)
- `db_mode: persistent` leaves DB file at artifact directory after run
